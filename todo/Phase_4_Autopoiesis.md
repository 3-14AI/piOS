# 5. Фаза IV: Архитектура Самодописывания (Autopoiesis)

**Цель фазы:** Замкнуть цикл разработки. Система должна уметь читать свой исходный код, модифицировать его, компилировать, верифицировать и обновлять себя.

Это наиболее экспериментальная часть.

**Технологический стек:** Cranelift/Winch, WASM Encoder, Verus (automated mode).

## Раздел 4.1: Компилятор как Сервис
Система должна содержать средства производства кода внутри себя.

- [ ] **WP-031 WASM Compiler Backend (Cranelift)**
  - *Описание:* Портирование Cranelift (или более легкого winch) в виде WASM-компонента. Это позволяет ОС компилировать промежуточное представление (IR) в машинный код внутри самой ОС.
  - *Сложность:* Высокая. Требуется адаптация под окружение ядра без доступа к файловой системе хоста.

- [ ] **WP-032 Rust Frontend (DSL to MIR)**
  - *Описание:* Реализация упрощенного фронтенда Rust или DSL, который ИИ может легко генерировать. Этот инструмент должен транслировать код в MIR (Mid-level IR) или сразу в WASM, минуя тяжеловесный LLVM.

- [ ] **WP-033 Сервис Автоматической Верификации (Verus Service)**
  - *Описание:* Интеграция решателя SMT (например, Z3) в ОС. Когда ИИ генерирует код, этот сервис запускает верификацию.
  - *Критично:* Если верификация не проходит, код отбрасывается. Это главный механизм безопасности "самописывания". ИИ может "галлюцинировать", но верификатор не пропустит некорректный код в ядро.

- [ ] **WP-034 Промпт-инжиниринг для Генерации Кода**
  - *Описание:* Создание системных промптов и контекста (RAG), обучающих LLM писать код, совместимый с Verus (с аннотациями requires, ensures). LLM должна знать API ядра и специфику доказательств.

- [ ] **WP-035 Механизм Горячей Замены (Hot-Reload)**
  - *Описание:* Логика безопасной замены компонента. 1. Приостановить старый компонент. 2. Сериализовать его состояние (State). 3. Инициализировать новый компонент. 4. Десериализовать состояние. 5. Переключить таблицу вызовов.
  - *Верификация:* Доказать функцию миграции состояния (State Transfer Function) — данные не теряются при обновлении.

## Раздел 4.2: Циклы Саморазвития

- [ ] **WP-036 Телеметрия и Петля Обратной Связи**
  - *Описание:* Система мониторинга производительности (cache misses, lock contention). Эти данные подаются на вход ИИ-агенту "Оптимизатор". Это дает ИИ причину для переписывания кода.

- [ ] **WP-037 Пайплайн Генерации Драйверов**
  - *Описание:* Сценарий: ОС обнаруживает неизвестное USB-устройство -> Агент ищет спецификацию в сети (через MCP) -> Генерирует код драйвера -> Verus верифицирует -> Драйвер загружается.

- [ ] **WP-038 Логика Самокоррекции (Self-Correction)**
  - *Описание:* Реализация цикла "Retry". Если WP-033 (Верификация) выдает ошибку, лог ошибки скармливается обратно LLM для исправления кода. Это превращает ошибку компилятора в обучающий сигнал.

- [ ] **WP-039 Синтез Регрессионных Тестов**
  - *Описание:* ИИ должен генерировать не только код, но и доказательства (proofs) и тесты. Разработка фреймворка, где генерация реализации и спецификации идет параллельно (Co-generation).

- [ ] **WP-040 Blue-Green Обновление Ядра**
  - *Описание:* Механизм запуска нового и старого компонента параллельно (shadow mode) для сравнения поведения на реальных данных перед окончательным переключением.
