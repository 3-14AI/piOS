3. Фаза II: WebAssembly User Space и Модель Драйверов Цель фазы : Реализовать пространство пользователя, где вместо нативных бинарных файлов (ELF/PE) используется WebAssembly. Это ключевой момент для реализации "самонаписания" (Self-coding) в будущем, так как WASM обеспечивает безопасную песочницу для сгенерированного ИИ кода. Технологический стек : Wasmtime (no_std fork), WASI (WebAssembly System Interface), WIT (WASM Interface Types). ## WP-012 Верификация песочницы памяти WASM

Описание : WASM гарантирует безопасность памяти через проверку границ (bounds checking). Верификация : Необходимо верифицировать "клей" (glue code) между рантаймом WASM и памятью ядра. Доказать в Verus, что функция memory.read/write рантайма никогда не выйдет за пределы выделенного региона WasmMemory, даже если байт-код скомпрометирован. Это защита от побега из песочницы (sandbox escape).

## WP-013 Реализация WASI-Core (Syscalls)

Описание : Реализация стандартов wasi-io, и Критерии верификации wasi-filesystem поверх примитивов ядра (IPC из WP-009). Верификация : Проверка соответствия спецификации WASI. Доказательство того, что файловый дескриптор, переданный в WASI- вызов, соответствует валидному объекту ядра и правам доступа (Capabilities).

## WP-014 Динамический Линковщик Компонентов (Linker)

Описание : Реализация механизма загрузки и связывания WASM- компонентов согласно Component Model. Это позволяет "подключать" драйверы как плагины. Важность : Это основа для Self-coding. Система сможет выгрузить старый компонент драйвера и загрузить новый, сгенерированный ИИ, на лету.

## WP-015 Система Капабилити (Capabilities System)

Описание : Реализация таблицы ресурсов (Resource Table) для каждого компонента. Доступ к железу (портам ввода-вывода, памяти DMA) выдается только через типизированные хендлы. Верификация : Доказать невозможность подделки хендла (handle forging). Компонент не может создать хендл "из воздуха", он должен получить его от родителя или ядра. ## WP-016 Драйвер VirtIO-Blk (WASM)

Описание : Реализация драйвера блочного и Критерии верификации устройства (диска) на Rust, компилируемого в WASM. Верификация : Доказательство корректности работы с кольцевыми буферами (Virtqueue). Инварианты: индексы avail и used колец всегда находятся в допустимых пределах, предотвращая переполнение буфера при обмене данными с гипервизором/железом.

## WP-017 Драйвер VirtIO-Net (WASM)

Описание : Сетевой драйвер. Верификация : Формализация обработки пакетов. Доказательство отсутствия утечек памяти при передаче пакетов (packet buffers) между сетевой картой и сетевым стеком.

## WP-018 PCI Bus Enumerator

Описание : Логика сканирования шины PCI и загрузки соответствующих WASM- драйверов. Верификация : Доказательство корректности парсинга конфигурационного пространства PCI. Гарантия того, что драйвер загружается только для устройства с совпадающим VendorID/DeviceID.

## WP-019 Абстракция DMA (Safe DMA)

Описание : Безопасный интерфейс для Direct Memory Access. Поскольку WASM не имеет прямого доступа к физ. памяти, ядро должно предоставлять специальные буферы. Верификация : Доказать, что регион DMA памяти "закреплен" (pinned) и не может быть перемещен или освобожден, пока устройство выполняет операцию ввода-вывода. и Критерии верификации

## WP-020 Виртуальная Файловая Система (VFS)

Описание : Компонент, объединяющий различные ФС. Верификация : Доказательство отсутствия циклов в графе директорий и корректности блокировок (rw-locks) при одновременном доступе к файлам.
